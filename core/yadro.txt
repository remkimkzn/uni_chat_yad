Здесь все жестко 
что ядро делает и чего нет
Принципы 
1.  Не знает про мессенджеры
2.  Не рендерит интерфейс
3.  Не хранит долгосрочные данные
Знает только
  Что есть полььзователь, канал, сообщение, сценарий/логика, состояние диалога
Контракт ядра с внешним миром 
    2 универсальных формата
Вход в ядро (от адаптера):
json
{
    "user_id": "string",
    "channel": "string",    // telegram, email
    "session_id": "string", // опционально
    "message": {
        "type": "text",
        "content": "Привет, хочу записаться",
        "metadata": {}
    },
    "context": {
        "bot_id": "string",
        "tenant_id": "string"
    }
}

Выход из ядра (к адаптеру):
{
    "user_id": "string",
    "channel": "string",
    "responses": [
        {
            "type": "text",

            "actions": [
                {
                    "type": "button",

                    "label": "Выбрать время",

Самое важное в ядре - смогут ли написать адаптер не трогая ядро
чем мало знает ядро тем лучше

Ядро кристально чистое - сделаем пошаговую цепочку ядра 
чтобы было стабильным, расширяемым, безопасным
1.  Адаптер приводит все к единому формату и вызывает endpoint ядра:
    /core/message
        Пакет состоит: user_id, channel, bot_id, message{type,content,metadata}, 
        timestamp, correlation_id.
    Ядро не знает что это telegram, email
это просто канал
2.  Message Router принимает пакет и делает  только
        Проверяет валидность формата.
        Определяет, какому боту принадлежит сообщение.
        Передает сообщение в Logic Engine
    Router  НЕ читает текст, НЕ анализирует смысл, НЕ знает сценариев
    Он просто маршрутизатор
3.  State Manager загружает состояние диалога 
    Ключ state_key = bot_id + user_id
    Он достает
        текущий шаг сценария
        временные данные
        контекст ИИ
        историю (короткую)
        флаги (например, ожидаем ввод телефона)
    Состояние - абстракное
4.  Logic Engine получает сообщение(пакет) + состояние
    Вот наши мозги важна строгость 
    Определяем тип входа 
        текст
        действие (кнопка)
        файл
        системное событие
    Находит активный сценарий  - думаю декларативный (JSON/YAML)
    Выполняет шаг сценария
        В зависимости от текущего шага
        задает вопрос
        ждет ввод
        вызывает функцию
        вызывает AI
        переходит на другой шаг
        завершает сценарий
    Обновляет состояние 
    формирует
        {
            state_patch = {
                step: ...,
                data: ...,
                flags: ...,
            }
        }
    Формирует универсальные ответы
    Не делает кнопки а формирует абстрактные ответы:
        {
            type: "action",
            id: "choose_time",
            label: "Выбрать время"
        }
5.  Responce Builder - приводит к универсальному формату
    Конструктор пакетов:
        нормализует типы
        проверяет структуру
        упаковывает в единый формата
    Он не знает как это будет выглядить в мессенджере
            {
                "responces": [...],
                "state_patch": {...}
            }
6.  State Manager сохраняет обновленное состояние 
    применяет state_patch:
        обновляет шаг
        обновляет данные
        обновляет контекст
        сохраняем постгрес
7.  Message Router возвращает универсальный ответ адаптеру
        {
            user_id,
            channel,
            responses,
            metadatqaa
        }
Адаптер сам превращает это в кнопки, форматы и т.д.
